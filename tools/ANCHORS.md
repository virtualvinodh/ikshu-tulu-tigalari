# Anchor Reference

Auto-generated by `generate_anchor_doc.py` from the current UFO + `glyph_classification.json` + `variant_registry.json` - re-run after `generate_anchors.py` to keep this in sync. Documents every GPOS anchor pair `generate_anchors.py` writes: which glyphs get it and the exact algorithm used to compute its x/y.

## Overview

| Anchor pair | Used for |
|---|---|
| `top` / `_top` | Repha, Svarita |
| `bottom` / `_bottom` | Anudatta, Gemination mark, Vocalic-L, Vocalic-LL |
| `topright` / `_topright` | Virama (USE "Bindu" override behavior) |
| `bottomright` / `_bottomright` | Vowel Sign U/UU/Vocalic-R/Vocalic-RR (fallback mark-attachment path, used when no dedicated ligature glyph exists) |

All anchors are re-runnable/idempotent: `upsert_anchor()` updates an existing anchor of the same name in place rather than skipping or duplicating it, so re-running `generate_anchors.py` after a GAP constant or algorithm change retunes every affected glyph with no manual cleanup.

## `bottomright` / `_bottomright`

This is the most-iterated-on anchor pair in the font (fallback mark-attachment path for U/UU/Vocalic-R/Vocalic-RR when GSUB leaves no dedicated per-consonant ligature glyph to fall back on) and has three distinct algorithms depending on glyph shape.

### Base side (`bottomright`)

**Default algorithm (`find_ink_bottom_right`)** - not bbox-based: scans the glyph's actual outline (including through components) for the point that maximizes the blended score `(x-xMin)/width + (1-(y-yMin)/height)` - i.e. simultaneously furthest right AND furthest down - so the fallback vowel sign visually touches the consonant's ink instead of floating off an empty bbox corner.

### Mark side (`_bottomright`)

**Default algorithm (`find_ink_top_right`)** - the mirror-image blended ink search: maximizes `(x-xMin)/width + (y-yMin)/height` - furthest right AND furthest **up**. This is where these marks visually begin (matching the base glyph's own furthest-right-furthest-down point, which is where they're meant to attach). An earlier leftmost-point approach was tried and rejected (2026-07-22): U/UU are round/circular enough that their true leftmost point sits at the vertical middle, nowhere near the visual start of the stroke.

**Override algorithm (`find_topmost_point`, via `MARKS_BOTTOMRIGHT_TOPMOST_OVERRIDE`, 10 glyphs):** simply the glyph's own topmost outline point (max y), at that point's true x - not forced to bbox center, but for a round dot shape this naturally lands near the horizontal center anyway (it's the shape's own "12 o'clock", where quadrant 1 starts going clockwise). Confirmed by the project owner (2026-07-22) that these specific glyphs read as plain round dots, where the right-biased blended score was visibly wrong:

`uMatra-tutg`, `uMatra-tutg.alt2`, `uMatra-tutg.alt3`, `uMatra-tutg.alt4`, `uMatra-tutg.alt5`, `uMatra-tutg.alt6`, `uMatra-tutg.below`, `uuMatra-tutg`, `uuMatra-tutg.alt4`, `uuMatra-tutg.below`

## Source functions (in `generate_anchors.py`)

| Function | Formula |
|---|---|
| `base_pass(pop, name, gap, x_from, y_edge)` | bbox edge +/- gap, x = center or edge |
| `mark_pass(names, name, x_from, y_edge)` | bbox edge, x = center/left/right, no gap |
| `find_ink_bottom_right(glyph, font)` | ink search, max((x-xMin)/w + (1-(y-yMin)/h)) - furthest right + furthest down |
| `find_ink_top_right(glyph, font)` | ink search, max((x-xMin)/w + (y-yMin)/h) - furthest right + furthest up |
| `find_lowest_point(glyph, font)` | ink search, true minimum-y outline point |
| `find_topmost_point(glyph, font)` | ink search, true maximum-y outline point |
